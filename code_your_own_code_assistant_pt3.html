Code Your Own Code Assistant Part 3

Part 2 defines the basic file system tools that the assistant will use. The full source is on GitHub.

In Part 3, now that we have tools wired up, we will use the model itself to extend our assistant — asking it to add new functionality to its own source code.

Why Do This

The interesting thing about having file system tools is that the assistant can now read and write its own source file. Once you realize that, a strange loop opens up: you can prompt the assistant to add new features to itself.

This is worth doing for a few reasons:

It’s a great way to understand how tools work in practice. Building the tooling yourself, rather than reaching for an existing framework, forces you to understand what’s happening at each step — how the model signals it wants to call a tool, how you dispatch to the right function, how the result gets fed back into the conversation. That knowledge is hard to get by reading documentation alone.

It’s a form of bootstrapping. Once the assistant can modify itself, adding new capabilities is just a matter of describing what you want. You’re no longer writing code by hand — you’re directing the model to write it for you, and then immediately using the result.

It’s a useful pattern to know. Self-modifying or self-extending systems show up in agentic workflows all the time. Starting with a simple example makes the underlying mechanics obvious.

Tools

We can do the same thing with tools. Rather than writing a new tool from scratch, we can point the assistant at a specification file and ask it to implement the tool from that description.

Since we are building on top of Anthropic’s Claude, we can borrow a SKILL.md file from Clawhub. Clawhub is a community hub for sharing skill files — structured markdown documents that describe how to use a particular tool or technique. Claude Code uses these natively; our custom assistant can use them too, as a specification to build from.

A word of caution: be careful when using skill files from Clawhub. Community-contributed files can be malicious — crafted to exfiltrate data or execute unintended commands. Read any skill file you plan to use before handing it to your assistant, and use the “Hide suspicious” filter when browsing the hub.

For this example I used the curl skill from Clawhub. After reviewing it, I dropped it into skills/curl/SKILL.md in the project directory.

Now prompt the assistant to build a tool from it:


🧑‍💻 The skills/curl/SKILL.md file explains how to use curl, please create a tool in main.py that will utilize it, this tool will be used when a page is retrieved from a url or when data is fetched from a API

🦀: Here is a full breakdown of everything that was added and why each decision was made.

### 4. Wired into `chat()` and `TOOLS`

Caveats

This is a proof of concept, not production code. The assistant-generated additions were accepted largely as-is — no error handling, no input validation, no guardrails on what files can be read or written. That’s fine for a local experiment where you control the prompts, but it would be a problem anywhere else.

A few things worth addressing before taking this further:

  • Error handling. If a tool fails — file not found, network timeout, bad API response — the assistant currently gets no useful feedback. Wrapping tool functions in try/except and returning structured error messages makes the model dramatically better at recovering gracefully.
  • Sandboxing. The write_file tool will happily overwrite any path the model provides. In practice you’d want to restrict it to a working directory, or at minimum prompt the user before writing.
  • LLM-generated code needs review. The model’s self-modifications worked correctly in these examples, but that won’t always be true. Treat the output the same way you’d treat a pull request from someone you don’t fully trust: read it before you run it.

Further Ideas

A few directions worth exploring from here:

  • Add other model providers. The main loop is generic enough that swapping in OpenAI or Gemini means adding a new client and a new branch in chat. The rest of the code doesn’t change.
  • Add local models via Ollama. Ollama exposes an OpenAI-compatible API, so you can point the client at localhost and run models entirely offline. Useful for sensitive codebases.
  • Add a file editing tool. Rather than overwriting entire files, a diff-based write tool would let Claude make targeted edits — less likely to accidentally clobber unrelated code.
  • Add a git tool. A tool that can run git diff, git log, or git status gives the assistant meaningful context about what’s changed and why.

The real power here is that you can shape the assistant to fit a specific use case — a domain-specific coding tool, a documentation writer, a test generator — rather than accepting a general-purpose default. And once you have file system tools, you can use the assistant itself to do the shaping.

Conclusion

Over these three parts we’ve gone from a basic conversation loop to an assistant that can read and write files, call external APIs, switch models at runtime, and extend itself by writing its own tools. None of the individual pieces are complicated — the Anthropic API is well-designed and the patterns are consistent — but putting them together reveals something genuinely useful.

The self-modification angle is the part I find most interesting. Once the assistant can act on its own source code, the workflow shifts: instead of writing features by hand, you describe what you want and let the model implement it. That’s not a replacement for understanding what’s happening under the hood — you still need to read the output, test it, and decide whether it’s correct — but it changes the shape of the work.
